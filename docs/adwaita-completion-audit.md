# OmniUI Adwaita Backend Completion Audit

Date: 2026-05-09

## Objective Restatement

Build a native Adwaita/GTK backend for OmniUI that lowers the OmniUI semantic/SwiftUI-compatible view model into native widgets alongside the existing notcurses backend. The supported scope is the BrowserView-required SwiftUI subset, with a kitchen sink demo, focused tests, and Computer Use verification. The broader "full SwiftUI drop-in replacement" wording is not treated as complete unless the implementation proves literal SwiftUI parity.

## Known Gaps

- This is not literal full SwiftUI parity. The supported contract is the BrowserView-required subset.
- Incremental reconciliation handles stable native leaf updates and localized structural subtree replacement. Structural changes spanning multiple stable ancestors still rebuild the GTK root, with scroll offset preservation.
- Computer Use does not enumerate GTK child controls as macOS accessibility elements in this environment; child behavior is verified through visible screenshot deltas.
- Raw GTK executables are not reliably discoverable by Computer Use as macOS apps; the kitchen sink and facade smoke verification therefore use minimal generated `.app` bundles in `scripts/run-kitchensink-adwaita-app.sh` and `scripts/run-omniui-adwaita-smoke-app.sh`.
- Full local `env -u ANTHROPIC_API_KEY -u OPENAI_API_KEY -u GEMINI_API_KEY -u GOOGLE_API_KEY -u GROQ_API_KEY -u CEREBRAS_API_KEY xcrun swift test --no-parallel` passed on 2026-05-09 with 726 tests in 121 suites after the latest Adwaita audit, state-wrapper, SwiftData delete, semantic toolbar preservation, stable `ForEach(..., id:)` list-row state, and Computer Use continuity additions. Live-provider E2E checks are still gated by API-key availability; the no-provider run verifies the local suite without rate-limit-prone external calls.
