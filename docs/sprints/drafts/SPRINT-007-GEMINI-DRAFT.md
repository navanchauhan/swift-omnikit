# Sprint 007: The Agent Product Runtime

## Overview

Sprint 007 transforms the foundational agent fabric established in Sprint 006 into a fully functional product runtime. It establishes the Root Agent as the exclusive user-facing persona—a "chief-of-staff" that brokers all interactions, accepts missions, and orchestrates execution. This sprint introduces a multi-user and workspace-aware foundation, ensuring strict data and context isolation. 

We will ship Telegram as the first external ingress, keeping the underlying architecture strictly transport-agnostic. Additionally, `OmniAIAttractor` will be formally integrated as the structured workflow engine for complex worker executions (e.g., `plan -> implement -> validate` loops), while preserving lightweight execution paths for atomic tasks. Finally, this sprint hardens the distributed system by enabling recursive delegation, routing all human-in-the-loop (HITL) approvals through the root, and closing the gap on remote artifact transport to support remote validation and judging.

## Use Cases

1. **Direct Mission Assignment:** A user sends a natural language request to the Root Agent via a Telegram Direct Message. The Root Agent interprets the mission, plans the execution, and delegates sub-tasks to available workers.
2. **Shared Workspace Collaboration:** Multiple users in a shared Telegram group interact with the Root Agent. The system recognizes the group as a distinct workspace, isolating its context, active missions, and artifacts from other groups or direct messages.
3. **Root-Brokered Approvals:** A remote worker executing an `OmniAIAttractor` pipeline requires user permission to modify a production system. The worker pauses and requests approval. The Root Agent surfaces this request to the user in the original Telegram thread. The worker resumes only when the user approves via the Root Agent.
4. **Structured Worker Execution:** The Root Agent delegates a complex refactoring task. The assigned worker utilizes an `OmniAIAttractor` pipeline to systematically plan the changes, implement them, and validate the compilation, reporting the final result back to the Root.
5. **Recursive Delegation & Supervision:** A worker realizes a task requires specialized capabilities (e.g., executing Python code). It delegates a sub-task to a child worker. The control plane tracks this lineage, enforcing depth limits and resource budgets.
6. **Remote Artifact Review:** A user asks the Root Agent to summarize a file generated by a remote worker. The Root Agent transparently fetches the artifact from the remote worker's local storage via the control plane and presents the summary.

## Architecture

The architecture builds upon the Sprint 006 HTTP mesh and SQLite-backed durable stores, introducing the following high-level components:

### Identity & Workspace Isolation
- **Foundation:** All durable stores (`ConversationStore`, `JobStore`, `ArtifactStore`) will be retrofitted with `workspace_id` and `user_id` dimensions.
- **Routing:** The control plane will enforce strict tenant boundaries. Context retrieval, task assignment, and artifact access will always be scoped to the active workspace.

### Ingress Gateway
- **Transport-Agnostic Core:** A new `InteractionBroker` in the control plane will manage incoming user events and outbound responses independently of the client platform.
- **Telegram Ingress:** A dedicated Telegram service (running as part of the control plane or a sidecar) will poll or receive webhooks from the Telegram Bot API. It normalizes Telegram updates into internal `UserMessage` events, mapping DMs to personal workspaces and Groups/Topics to shared workspaces.

### The Chief-of-Staff Runtime
- **Mission Coordinator:** The `RootAgentRuntime` transitions from managing raw tasks to managing "Missions". It acts as the ultimate supervisor, converting user intent into a directed graph of jobs.
- **Approval Broker:** All worker-initiated requests (human input, tool execution approvals) are routed up the hierarchy to the `RootAgentRuntime`, which formats them for the appropriate ingress channel and suspends the worker until a response is received.

### Attractor Integration & Execution
- **Task vs. Pipeline:** The `WorkerDaemon` will support multiple execution strategies. Simple tasks map directly to ACP execution. Complex, multi-step tasks invoke `OmniAIAttractor` pipelines (e.g., `ManagerLoopHandler`).
- **Lineage Tracking:** `TaskRecord` will support `parent_task_id` to build a durable tree of recursive delegations. The control plane will enforce max-depth policies to prevent runaway subagent loops.

### Remote Artifact Transport
- **Artifact Mesh Protocol:** Extension of the HTTP mesh to allow the control plane to request artifact blobs from remote worker nodes. This enables the Root Agent or Judge agents to inspect files generated on entirely different hosts.

## Implementation

The implementation is phased to ensure stability of the control plane while introducing user-facing features.

### Phase 1: Identity and Ingress Foundation
1. **Schema Migration:** Update SQLite schemas in `OmniAgentMesh` to include `workspace_id` and `user_id` across conversations, jobs, tasks, and artifacts.
2. **Ingress Abstraction:** Implement the `InteractionBroker` and define the internal `UserEvent` and `AgentResponse` models in `TheAgentControlPlane`.
3. **Telegram Integration:** Implement the Telegram ingress gateway. Map Telegram Chat IDs to Workspace IDs and User IDs to internal representations. Support parsing DMs and basic group mentions.

### Phase 2: The Chief-of-Staff Root
1. **Mission Orchestration:** Refactor `RootAgentRuntime` and `RootOrchestratorProfile` to operate on higher-level user intents.
2. **Approval Brokering:** Implement the routing of `WaitHumanHandler` requests from workers through the mesh to the Root Agent, and present them via the `InteractionBroker`.
3. **Conversation Management:** Ensure the Root Agent maintains coherent conversational context scoped strictly to the current workspace.

### Phase 3: Worker Execution and Attractor
1. **Attractor Wiring:** Integrate `OmniAIAttractor`'s `PipelineEngine` into `WorkerExecutorFactory` for tasks flagged as requiring structured workflows.
2. **Recursive Delegation:** Update `TaskRecord` and `ChildWorkerManager` to support and track durable lineage. Implement supervision bounds (budget, depth) in the `RootScheduler`.
3. **Remote Artifacts:** Implement the mesh endpoints in `HTTPMeshServer`/`HTTPMeshClient` for artifact retrieval. Update `ArtifactStore` to handle remote fetch resolution.

## Definition of Done

- Multi-user and workspace dimension added to all durable stores; tests prove zero cross-workspace data leakage.
- Telegram ingress is fully functional: a user can message the bot, start a mission, and receive status updates.
- The Root Agent operates as the sole user-facing persona, mediating all worker interactions.
- Workers can successfully execute a multi-step `OmniAIAttractor` pipeline (`plan -> implement -> validate`).
- Workers can recursively delegate sub-tasks, with the control plane enforcing a maximum recursion depth.
- A worker requiring human approval pauses execution; the Root Agent successfully requests and receives this approval via Telegram, and the worker resumes.
- The Root Agent can retrieve and read an artifact generated by a remote worker via the HTTP mesh.
- All code compiles under Swift 6 strict concurrency with no warnings.

## Risks & Mitigations

| Risk | Mitigation |
| :--- | :--- |
| **Workspace Leakage:** Bugs in context retrieval could expose one user's data to another. | Implement strict SQLite views or query wrappers that require a `workspace_id` context parameter for all data access. Backed by rigorous integration tests. |
| **Telegram Group Noise:** The bot might respond to every message in a shared group, causing spam. | Implement strict mention-parsing (`@botname`) for group routing. DMs default to implicit interaction. |
| **Recursive Runaways:** A poorly prompted worker creates an infinite loop of sub-tasks. | Enforce a hardcoded `max_depth` and `max_subtasks_per_job` in the `RootScheduler`. Terminate and alert on violations. |
| **Approval Deadlocks:** A worker waits for an approval that is never seen or acted upon by the user. | Implement timeouts on approval requests at the worker level, failing the task gracefully if the timeout is reached. |
| **Artifact Payload Size:** Fetching large artifacts (e.g., binaries) from remote workers could crash the control plane. | Implement streaming or strict size limits for the Artifact Mesh Protocol. |

## Security Considerations

- **Authentication & Ingress:** The Telegram ingress relies on Telegram's bot token validation. If a generic HTTP API is added later, it must require robust authentication.
- **Tenant Isolation:** The most critical security boundary is the `workspace_id`. All database queries must be audited to ensure they cannot bypass this filter.
- **Execution Boundaries:** Remote workers executing arbitrary code (via Attractor or standard tools) must remain sandboxed (e.g., Docker/macOS sandboxes established in previous sprints). The control plane must never execute worker payloads itself.

## Dependencies

- **Sprint 006:** Requires the stable `RootScheduler`, `WorkerDaemon`, HTTP mesh, and SQLite backing stores.
- **OmniAIAttractor:** Requires the existing `PipelineEngine` and `WaitHumanHandler` primitives.
